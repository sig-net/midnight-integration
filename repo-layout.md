# Repo layout proposal

## Naming

Proposed repo name: **`signet-midnight-demo`** (currently `midnight-erc20-vault`).

Rationale: the repo's mission is "demonstrate sig-net cross-chain interactions from Midnight";
the ERC20 vault is only the *first* example. Alternative if we expect several independent
examples to accumulate: `signet-midnight-examples`.

Package scope follows the repo name: `@signet-midnight-demo/*` (private, never published —
the scope only has to be unambiguous, not pretty).

## Scripts (root package.json)

Plain npm workspaces — no turbo/nx at this scale. npm does **not** topologically sort
`--workspaces` runs, so root scripts name workspaces explicitly in dependency order:

```jsonc
{
  "scripts": {
    "compact":          "npm run compact -w packages/vault-contract -w packages/signature-responses-contract",
    "build":            "npm run build -w packages/lib -w packages/signet-midnight -w packages/vault-contract -w packages/signature-responses-contract -w packages/deploy -w packages/testing",
    "test":             "npm run test --workspaces --if-present",
    "stack:up":         "docker compose -f scripts/stack/standalone.yml up -d && scripts/stack/wait-for-stack.sh",
    "stack:down":       "docker compose -f scripts/stack/standalone.yml down",
    "test:integration": "npm run test -w packages/integration-tests",
    "deploy:vault":     "npm run deploy -w packages/vault-contract",
    "deploy:responses": "npm run deploy -w packages/signature-responses-contract",
    "deploy:usdc":      "npm run deploy -w packages/evm",
    "response-server":  "npm run start -w packages/response-server",
    "ui":               "npm run dev -w packages/ui"
  }
}
```

This gives both modes: `stack:up && compact && build && test:integration` as a flow, or any
piece on its own. If ordering maintenance ever becomes painful, adopt turbo then — not now.

## Decisions taken

- **One package per contract** (no contract/sdk split). compactc output lands in the same
  package it came from; `src/index.ts` is the curated export surface. A split becomes worth
  revisiting only if a contract package is ever published to npm.
- **`managed/` is generated, never committed** — gitignored per contract package. CI runs
  `compact` before `build`, so the compact toolchain version must be pinned in CI.
- **Shared plumbing lives once, in `lib`.** No per-package copies of config/wallet/providers.
- **Generic deployer kept** (midday precedent proves the pattern), but each contract package
  owns a thin `deploy.ts` entrypoint so contract-specific bits (managed path, initialise call)
  stay next to the contract.
- **`signet-midnight` is a first-class package**, not buried in a CLI. It should lean on
  signet.js wherever signet.js already covers the ground (EVM tx preparation, address
  derivation) and only own what is Midnight-specific.
- **Flat `packages/`** — no `apps/` split. One UI in a demo repo doesn't justify the ceremony.
- **Integration-test ordering via global setup**, not numbered test files; stack lifecycle via
  `scripts/`, not from inside vitest.
- **The websocket response path is purged, no fallback.** All responses flow through
  signature-responses-contract; clients poll it.
- **The fakenet response server moves in** as `packages/response-server` — only the Midnight
  slice of solana-signet-program's `clients/response-server` (MidnightMonitor + the
  handleMidnightSigningRequest path + EVM confirmation + Schnorr responder). The Solana, Bitcoin
  and Substrate machinery stays where it is. The extraction also deletes that repo's vendored
  snapshot of our signet helpers and compiled vault contract, which are already drifting.

## Migration hygiene

Delete on migration (and cover in `.gitignore` so they never return):

- `boilerplate/contract-cli/logs/`, `stack.log`, `proof-server.log`
- `boilerplate/contract-cli/midnight-level-db/`
- `boilerplate/contract-cli/reports/`
- `boilerplate/evm/artifacts/`, `boilerplate/evm/cache/`
- committed `managed/` output under `boilerplate/contract/src/managed/`
- `counter.compact.example`, `.DS_Store`

## Open questions

1. Repo name: `signet-midnight-demo` vs `signet-midnight-examples` — pick based on whether
   this repo stays one example or accumulates several.
2. Audit `packages/signet-midnight` against signet.js before the move: anything it
   re-implements (EVM tx prep, request-id derivation?) should import from signet.js instead;
   what remains is the candidate list for upstreaming a Midnight adapter.
3. signature-responses-contract stores `Map<Bytes<32>, Bytes<32>>`, but the payloads that
   currently travel over the websocket are bigger than 32 bytes: a signed EVM tx is
   variable-length, and the Schnorr claim response is announcement point + scalar + outputData.
   Do map values become hashes/commitments with data fetched elsewhere, or does the value type
   grow (e.g. multiple entries / larger cells)?
4. Once response-server lives here, does solana-signet-program drop its Midnight support
   entirely? Two maintained copies of MidnightMonitor would recreate the drift this move is
   meant to kill.
